# Feature Specification: 로그인 및 기본 프로필 관리

**Feature Branch**: `001-auth-profile`  
**Created**: 2026-04-21  
**Status**: Draft  
**Input**: User description: "Google OAuth 로그인 + 기본 프로필(이름, 이메일, 전화, 한 줄 소개) 생성·수정"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 구글 계정으로 서비스에 로그인 (Priority: P1)

사용자는 웹 서비스 랜딩 페이지에서 "Google로 시작하기" 버튼을 눌러 본인의 구글 계정으로 서비스에 로그인한다. 로그인이 완료되면 본인 전용의 대시보드(프로필 화면)로 이동한다.

**Why this priority**: 로그인이 가능해야 사용자가 본인의 이력 데이터를 소유·격리하여 저장할 수 있다. 이후 모든 기능(프로필 편집, 익스텐션 자동 완성 등)은 "인증된 사용자"를 전제로 하므로 가장 먼저 동작해야 한다.

**Independent Test**: 새 구글 계정으로 사이트에 접속해 "Google로 시작하기"를 누르고 동의 화면을 통과하면, 본인 화면에 진입하고 다른 사용자 데이터에는 접근할 수 없다는 사실을 수동 확인으로 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 비로그인 상태로 랜딩 페이지를 열었을 때, **When** "Google로 시작하기"를 누르고 구글 동의 화면을 통과하면, **Then** 본인 이메일이 표시된 프로필 화면으로 이동한다.
2. **Given** 이미 로그인된 브라우저에서 사이트를 다시 열면, **When** 랜딩 페이지에 접근하더라도, **Then** 자동으로 본인 프로필 화면으로 리다이렉트된다.
3. **Given** 로그인된 상태에서 "로그아웃"을 누르면, **When** 다시 보호된 페이지(예: `/profile`)로 접근하려 하면, **Then** 랜딩 페이지로 돌려보낸다.

---

### User Story 2 - 기본 프로필 생성·수정 (Priority: P1)

최초 로그인 직후 빈 프로필이 자동 생성되어 있고, 사용자는 이름·이메일·전화번호·한 줄 소개 네 가지 필드를 편집·저장할 수 있다. 이메일은 구글 계정 값으로 초기 채워져 있다.

**Why this priority**: 이력 관리 서비스의 최소 단위가 "프로필"이다. 이 데이터가 저장 가능해야 이후 모든 이력 항목(경력·학력·스킬 등)이 붙을 수 있고, 익스텐션 자동 완성의 최소 동작(이름·이메일 채우기)도 실행 가능해진다.

**Independent Test**: 로그인한 사용자가 네 필드를 입력·저장하고 페이지를 새로고침하거나 재로그인해도 값이 그대로 유지되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 방금 최초 로그인한 사용자가 프로필 화면에 들어오면, **When** 화면이 로드되면, **Then** 이메일 필드에는 구글 계정 이메일이 이미 채워져 있고 이름에는 구글 계정의 표시 이름이 기본값으로 들어가 있다.
2. **Given** 프로필 화면에서 네 필드를 수정하고 "저장"을 누르면, **When** 서버 응답을 받으면, **Then** "저장됨" 상태가 표시되고, 페이지를 새로고침해도 수정값이 유지된다.
3. **Given** 이름 또는 이메일을 비운 상태로 저장하려 하면, **When** "저장"을 누르면, **Then** 해당 필드에 검증 메시지가 표시되고 서버에는 전송되지 않는다.
4. **Given** 사용자 A가 프로필을 저장한 상태에서 사용자 B가 로그인하면, **When** 사용자 B가 프로필 화면을 열면, **Then** 사용자 B의 빈 프로필만 보이고 A의 값은 조회되지 않는다.

---

### Edge Cases

- 구글 동의 화면에서 사용자가 거부하거나 창을 닫으면: 랜딩 페이지로 돌아가 "로그인이 취소되었습니다" 알림을 보여준다.
- 로그인 중 네트워크 장애로 세션 수립이 실패하면: 오류 알림과 함께 다시 시도할 수 있는 버튼을 제공한다.
- 저장 중 네트워크 장애가 발생하면: "저장 실패" 상태를 표시하고, 입력 값은 로컬 폼에 그대로 유지한다.
- 전화번호·한 줄 소개는 선택 입력이다. 공란 저장을 허용한다.
- 동일 구글 계정으로 두 브라우저에서 동시에 수정 시: 마지막 저장이 서버의 최종값이 된다(낙관적 덮어쓰기). 충돌 UI는 v1에서 다루지 않는다.
- 이메일 필드는 구글 계정에서 받아온 값으로 고정한다. 사용자가 수정할 수 없다(연락 이메일이 필요해지면 별도 필드를 이후 버전에서 추가).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 랜딩 페이지에 "Google로 시작하기" 버튼을 노출해야 한다.
- **FR-002**: 시스템은 Google OAuth 2.0 인증을 통해 사용자를 식별할 수 있어야 한다.
- **FR-003**: 최초 로그인 시 시스템은 해당 사용자의 빈 프로필 레코드를 자동 생성해야 한다. 이메일은 구글 계정 값으로, 이름은 구글 표시 이름으로 초기화한다.
- **FR-004**: 로그인된 사용자는 프로필의 이름·전화번호·한 줄 소개를 편집·저장할 수 있어야 한다.
- **FR-005**: 이메일 필드는 읽기 전용이어야 한다.
- **FR-006**: 시스템은 로그인되지 않은 사용자가 보호된 경로(예: 프로필 화면)에 접근할 경우 랜딩 페이지로 리다이렉트해야 한다.
- **FR-007**: 시스템은 사용자 A가 사용자 B의 프로필을 조회·수정할 수 없도록 데이터 격리를 보장해야 한다.
- **FR-008**: 사용자는 언제든 로그아웃할 수 있어야 하며, 로그아웃 후 세션은 즉시 만료돼야 한다.
- **FR-009**: 시스템은 이름이 비었거나 형식이 올바르지 않은 저장 요청은 거부하고 사용자에게 오류 사유를 알려야 한다.
- **FR-010**: 저장 성공·실패는 사용자에게 시각적으로 피드백돼야 한다(성공 시 "저장됨" 배지, 실패 시 오류 메시지와 재시도 수단).

### Key Entities

- **User (사용자)**: 구글 계정을 통해 식별된 서비스 사용자. 핵심 속성: 내부 사용자 식별자, 구글에서 받은 이메일, 가입 시각, 최근 로그인 시각.
- **Profile (프로필)**: 사용자 1명당 정확히 1개 존재. 핵심 속성: 소유자(User 참조), 이름, 이메일(User의 이메일 복제/동기), 전화번호(선택), 한 줄 소개(선택), 마지막 수정 시각. 소유자 외에는 조회·수정 불가.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 신규 사용자가 랜딩 페이지 진입부터 로그인 완료(본인 프로필 화면 도달)까지 30초 이내에 마칠 수 있다.
- **SC-002**: 로그인된 사용자가 프로필 네 필드를 입력하고 저장하기까지 1분 이내에 완료할 수 있다.
- **SC-003**: 프로필 저장 후 페이지를 새로고침하거나 다시 로그인했을 때 100% 동일한 값이 표시된다.
- **SC-004**: 한 사용자가 저장한 프로필 값이 다른 사용자에게 절대 노출되지 않는다(0건 데이터 누출).
- **SC-005**: 저장 요청의 95% 이상이 사용자 관점에서 즉각적(2초 이내)으로 완료된 것처럼 느껴진다.

## Assumptions

- 사용자는 이미 구글 계정을 보유하고 있다. 구글 계정이 없는 사용자를 위한 별도 가입 플로우는 v1 범위 밖이다.
- 프로필은 한 사용자당 1개로 고정한다. 여러 버전/프리셋 프로필을 운영하는 기능은 v1 범위 밖이다.
- 이력의 상세 항목(경력·학력·스킬 등)은 본 스펙 범위 밖이며, 별도 스펙에서 다룬다.
- 웹 익스텐션은 본 스펙 범위 밖이다. 단, 익스텐션이 추후 읽을 수 있도록 프로필 데이터는 사용자 토큰으로 접근 가능한 위치에 저장한다.
- 모바일 전용 레이아웃은 본 스펙 범위 밖이지만 데스크톱 레이아웃이 모바일에서도 읽을 수 있어야 한다(반응형 최소 대응).
- 다국어 지원은 범위 밖이다. UI 언어는 한국어로 고정한다.
- 개인정보 취급 방침·약관 페이지의 법적 검토는 v1 이후 별도 트랙으로 진행한다(MVP에서는 플레이스홀더 링크만 배치).
