# Feature Specification: 경력 관리 (Work Experience CRUD)

**Feature Branch**: `003-work-experience`  
**Created**: 2026-04-22  
**Status**: Draft  
**Input**: User description: "경력 CRUD — 여러 회사의 경력 이력을 추가·수정·삭제·정렬하여 관리"

## Clarifications

### Session 2026-04-22

- Q: 경력 설명(`description`) 최대 글자 수 → A: **500 code points**. 한 경력당 3~5줄 bullet-point 수준의 간결한 서술. 자기소개(1000자)와 다른 역할 — 경력은 여러 개 쌓이므로 각각 짧게 유지해야 리스트 가독성이 살아있다.
- Q: 경력 항목 총 개수 상한 → A: **50건**. 실사용 관점에서 사실상 무제한 (컨설턴트·프리랜서 포함). UI 렌더 비용도 50건은 가상화 없이 감당.
- Q: 편집 UI 형태 → A: **인라인 확장**. 리스트 항목을 클릭하면 해당 카드가 편집 모드로 전환, 저장 후 다시 리스트 뷰로. Notion·Linear 스타일. 모달의 공격성·별도 페이지의 route 복잡도 회피 + 작성 중 리스트 전체 context 유지.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 경력 항목 추가 (Priority: P1)

사용자는 프로필 화면에서 "경력 추가" 버튼을 눌러 새 경력 항목을 생성한다. 회사명·역할·시작일은 필수, 종료일·설명은 선택. "현재 재직 중"인 경우 종료일을 비울 수 있다.

**Why this priority**: 경력은 이력서의 핵심 본문. 이 피처가 돌아가야 제품이 "이력 관리 서비스"라는 이름에 부합한다. 익스텐션 자동 완성의 가장 큰 타겟도 경력 필드.

**Independent Test**: 로그인한 사용자가 경력 0건 상태에서 시작 → "경력 추가" → 회사·역할·시작일 입력 → 저장 → 경력 리스트에 새 항목 1건 표시. 재로드 후에도 유지.

**Acceptance Scenarios**:

1. **Given** 로그인한 사용자가 경력 0건 상태, **When** "경력 추가"를 누르고 회사 `Acme Corp`·역할 `Senior Engineer`·시작일 `2023-01`을 입력하고 저장, **Then** 경력 리스트 상단에 해당 항목이 표시된다.
2. **Given** "현재 재직 중" 토글이 켜진 상태에서 새 경력을 추가하면, **When** 종료일 입력란이 비활성화되고 저장 가능, **Then** 리스트에 "2023-01 – 현재"로 표시된다.
3. **Given** 시작일만 입력하고 회사명을 비운 채 저장 시도, **When** "저장"을 누르면, **Then** 회사 필드에 검증 오류 표시 + 저장 차단.
4. **Given** 시작일이 종료일보다 나중인 값을 입력, **When** 저장을 누르면, **Then** 기간 필드에 오류 "종료일은 시작일 이후여야 합니다" 표시.

---

### User Story 2 - 경력 항목 편집 (Priority: P1)

사용자는 기존 경력 항목을 편집할 수 있다. 항목을 클릭(또는 편집 버튼)하면 편집 모드로 전환되고, 필드 수정 후 저장하면 업데이트된다.

**Why this priority**: 경력 정보는 시간에 따라 업데이트됨(종료일 추가, 역할 변경 등). 편집이 안 되면 삭제 후 재생성해야 해서 이탈 유발.

**Independent Test**: 경력 1건 존재 상태에서 편집 → 역할 필드만 변경 → 저장 → 해당 항목만 반영, 다른 필드는 그대로.

**Acceptance Scenarios**:

1. **Given** 경력 1건 존재, **When** 해당 항목의 편집 버튼 클릭 → 역할을 `Lead Engineer`로 변경 → 저장, **Then** 리스트에 변경된 역할이 표시된다.
2. **Given** "현재 재직 중"으로 저장돼 있던 경력, **When** 편집 진입 후 종료일 `2024-06`을 입력하고 "현재 재직 중" 해제, **Then** "2023-01 – 2024-06"으로 표시.
3. **Given** 편집 중 아직 저장하지 않은 상태, **When** 페이지를 벗어나려 하면, **Then** 변경 사항 손실 경고 표시(001 beforeunload 패턴 재사용).

---

### User Story 3 - 경력 항목 삭제 (Priority: P2)

사용자는 더 이상 표시하고 싶지 않은 경력을 삭제할 수 있다.

**Why this priority**: P1은 아니지만 이력 관리상 반드시 필요한 기능. 잘못 등록한 항목·민감한 과거 경력 제거. (원래 P2로 표시했으나 상한 50건 맥락에서 삭제가 필수 수단이 되므로 실제 구현 우선순위는 P1에 준한다.)

**Independent Test**: 경력 2건 존재 → 1건 삭제 → 리스트에 1건만 남음 → 재로드 후 동일.

**Acceptance Scenarios**:

1. **Given** 경력 2건 존재, **When** 1번째 항목의 삭제 아이콘 클릭 → 확인 다이얼로그에서 "삭제" 선택, **Then** 해당 항목이 리스트에서 사라지고 DB에서도 제거된다.
2. **Given** 삭제 확인 다이얼로그, **When** "취소"를 누르면, **Then** 다이얼로그가 닫히고 항목은 유지.

---

### User Story 4 - 경력 정렬 표시 (Priority: P2)

경력 리스트는 **최신 경력이 위**에 오도록 정렬되어 표시된다.

**Why this priority**: 채용 관행상 역순 시간 정렬(reverse chronological)이 표준. 수동 드래그 정렬은 v1 범위 밖 (Assumptions 참조).

**Independent Test**: 경력 3건(시작일 2020, 2022, 2024)을 순서 상관없이 추가 → 리스트에 `2024 → 2022 → 2020` 순으로 표시.

**Acceptance Scenarios**:

1. **Given** 경력 3건(시작일 2020·2022·2024), **When** 프로필 화면에 진입, **Then** 리스트 순서는 `2024, 2022, 2020`.
2. **Given** "현재 재직 중" 경력(시작 2024) + 종료된 경력(2023-2024), **When** 리스트 로드, **Then** "현재 재직 중" 항목이 가장 위.

---

### Edge Cases

- **한 달 단위 vs 일 단위 날짜 입력**: 채용 이력 관습에 맞게 **월 단위(`YYYY-MM`)** 입력 권장. 일 단위까지 받으면 과도한 정밀도.
- **"현재 재직 중" 복수**: 동시에 여러 회사 재직(겸직)이 가능하다. 시스템적으로 복수 "current" 허용.
- **시작일 = 종료일**: 허용(1개월 단기 근무). 검증은 `종료일 >= 시작일`.
- **긴 설명 입력**: 카운터를 필드 우상단에 표시. 400자 이상부터 warn 색, 500자에서 danger + 추가 입력 차단(002 패턴 재사용, 80% 비율 룰).
- **회사명에 이모지/특수문자**: 허용(유효한 유니코드 문자열).
- **삭제 취소 불가**: v1에서는 hard delete. undo 패턴은 v2 이후.
- **빈 상태 (0건)**: "아직 경력이 없습니다. 첫 경력을 추가해보세요." 안내 + "경력 추가" CTA.
- **경력 저장 중 네트워크 장애**: 001의 inline 에러 블록 + 재시도 패턴 재사용.
- **동시 편집 (여러 탭)**: 낙관적 덮어쓰기 — 마지막 저장이 이김. 001 assumption 재사용.
- **총 경력 수 상한**: **50건**. 51번째 "경력 추가" 시도 시 알림 메시지 표시 + 버튼 비활성. 실제 이 한계에 도달하는 사용자는 극소수 예상.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 `work_experiences` 테이블에 사용자별 경력 이력을 저장해야 한다. 사용자 1명당 복수 경력 허용.
- **FR-002**: 경력 항목은 다음 필드를 가진다:
  - `company` (회사명, 필수, 1–100자)
  - `role` (역할/직책, 필수, 1–100자)
  - `start_date` (시작일, 필수, 월 단위 `YYYY-MM-01` 형태로 저장)
  - `end_date` (종료일, 선택. NULL = 현재 재직 중)
  - `description` (설명, 선택, **최대 500 code points** — `Array.from(str).length` 기준, 002-profile-summary 규칙 상속)
- **FR-003**: 사용자는 경력을 추가·편집·삭제할 수 있어야 한다.
- **FR-004**: 시스템은 경력 리스트를 **start_date 내림차순**(최신이 위)으로 반환한다. `end_date IS NULL`(현재 재직 중)인 항목은 같은 시작일 기준에서 더 위에 배치.
- **FR-005**: 삭제 요청 시 확인 다이얼로그를 표시하고, 사용자가 "삭제"를 확인하면 레코드를 즉시 제거한다(soft delete 아님).
- **FR-006**: RLS 정책으로 사용자 A는 사용자 B의 경력을 조회·수정·삭제할 수 없어야 한다(`profiles` 동일 패턴).
- **FR-007**: 빈 상태(경력 0건)에선 안내 메시지와 "경력 추가" CTA를 노출한다.
- **FR-008**: `end_date < start_date`인 데이터는 클라이언트·DB 양쪽에서 거부한다.
- **FR-009**: 편집/추가 중 저장되지 않은 변경이 있을 때 페이지 이탈 시도는 `beforeunload` 경고로 차단(001 FR-011 패턴 재사용).
- **FR-010**: 회사·역할·설명 필드는 WCAG 2.1 AA 준수(001 FR-012 패턴 재사용).
- **FR-011**: 동일 사용자가 **동시에 "현재 재직 중" 경력을 여러 개 보유**할 수 있어야 한다(겸직 허용).
- **FR-012**: 사용자당 경력 항목은 **최대 50건**까지 저장 가능. 상한 도달 시 "경력 추가" 버튼 비활성 + 안내 메시지 "최대 50건까지 등록 가능합니다".
- **FR-013**: 경력 편집은 **리스트 항목 인라인 확장** 방식으로 수행한다. 페이지 이동·모달 오버레이 없이 해당 카드 내에서 뷰 모드 ↔ 편집 모드를 전환.

### Key Entities

- **WorkExperience** (`public.work_experiences`): 사용자 1명당 0..N개 존재. 핵심 속성:
  - `id` — 고유 식별자(UUID)
  - `user_id` — 소유자(User 참조, FK → auth.users)
  - `company`, `role`, `start_date`, `end_date`, `description` — FR-002 참조
  - `created_at`, `updated_at` — 감사용 타임스탬프
  - 소유자 외에는 조회·수정·삭제 불가(RLS).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자가 새 경력을 추가하고 "저장"을 누른 뒤 **리스트에 반영되기까지 2초 이내**(p95).
- **SC-002**: 경력 N개(N≤20) 보유한 사용자의 프로필 페이지 초기 로드(LCP)는 **2.5초 이내**(p75, SC-007 of 001 상속).
- **SC-003**: 사용자 A의 경력은 사용자 B에게 0건 노출(RLS 검증, 수동 스모크 포함).
- **SC-004**: 시작일 > 종료일인 데이터는 클라이언트·DB 양쪽에서 100% 거부.
- **SC-005**: 편집 중 페이지 이탈 시도는 100% 경고 표시(`beforeunload`).
- **SC-006**: 경력 항목 편집/추가 폼을 **키보드만으로 완주** 가능(SC-006 of 001 상속).

## Assumptions

- 001-auth-profile, 002-profile-summary 완료 전제(로그인·프로필 저장 동작).
- 날짜 입력은 **월 단위**(`YYYY-MM`). 일 단위 UI는 범위 밖.
- 경력 항목에 대한 **파일 업로드(증빙서류 등)는 범위 밖**.
- 드래그 앤 드롭 수동 정렬·"고정(pin)" 기능은 범위 밖(자동 정렬만).
- 첨부 회사 로고·링크드인 매핑은 범위 밖.
- 개인정보 삭제 요청 시 경력 테이블도 CASCADE로 함께 제거(001 FR-013 상속).
- 설명(description)에 마크다운·서식 지원 없음. 평문 + 줄바꿈만.
